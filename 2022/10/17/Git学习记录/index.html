版本控制

本地版本控制

使用软件来记录文件的不同版本,提高工作效率降低手动维护版本的出错率

缺点

  • 单机运行,不支持多人协作开发
  • 版本数据库故障后,所有历史记录会丢失

集中化版本控制

基于服务器,客户端的运行模式

  • 服务器保存所有文件的更新记录

  • 客户端只保留最新的文件版本

  • 联网运行,支持多人协作开发

缺点

  • 不支持离线提交版本更新
  • 中心服务器崩溃后,所有人无法正常工作
  • 版本数据库故障后,所有历史更新记录会丢失

分布式版本控制

基于服务器客户端的运行模式

  • 服务器保存文件的所有更新版本
  • 客户端是服务器的完整备份,并不只保存最新版本
  • 联网运行,支持多人协作开发
  • 客户端断网后支持离线本地提交版本更新
  • 服务器故障后可使用任何一个客户端的备份进行恢复

GIT

关于git

Git是一个开源的分布式版本管理控制系统,是目前世界上最先进最流行的版本控制系统,可以高效地处理从很小到非常大的项目版本管理.

特性

  • 直接记录快照,而非差异比较
  • 近乎所有操作都是本地执行
差异比较

传统的版本控制系统(如SVN)是基于差异的版本控制,存储一组基本文件和每个文件随时间逐步累积的差异

  • 节省磁盘空间
  • 耗时低效
记录快照

在原有文件版本的基础上重新生成一份新文件,类似于备份.为了效率如果文件没有修改,则只保留一个链接指向之前文件

  • 占用磁盘空间
  • 版本切换高效,因为每个版本都是完整的文件快照

GIT中的三个区域

工作区

处理工作的区域

暂存区

已完成的工作的临时存放区域,等待被提交

Git仓库

最终的存放区域

Git中的三种状态

已修改 modified

表示修改了文件,但还没将修改的结果提交到暂存区

已暂存 staged

表示对已修改文件的当前版本做了标记,使之包含在下次提交的列表中(已修改并放入暂存区)

已提交 committed

表示文件已安全地保存在本地的Git仓库中

基本的Git工作流程

  1. 在工作区中修改文件
  2. 将你想要下次提交的更改进行暂存
  3. 提交更新,找到暂存区的文件,将快照永久性存储到Git仓库

Git的基本操作

获取Git仓库

  • 将尚未进行版本控制的本地目录转换为Git仓库
  • 从其他服务器克隆一个已存在的Git仓库
从现有目录初始化仓库
  • 在项目目录中通过鼠标右键打开’Git Bash’
  • 执行git init命令将当前目录转换为Git仓库
提交文件到仓库中
git add
  • 可以用来跟踪新文件
  • 把已跟踪的,且已修改的文件放到暂存区
  • 把有冲突的文件标为已解决状态
git add .

一次性将所有新增和修改过的文件添加到暂存区中

git status

查询当前文件目录下文件状态

  • $\textcolor{red}{modified(M)}$: 已被修改但还未放入暂存区
  • $\textcolor{green}{modified(M)}$:已被修改且放入暂存区
  • Untracked$\textcolor{red}{(??)}$:未被跟踪
  • $\textcolor{green}{D}$:当前文件已被标记为删除
git commit -m “ 描述信息”

将暂存区中记录的文件提交到Git仓库中进行保存

git commit -a -m “描述信息”

将所有已经跟踪过的文件暂存起来并一起提交到git仓库

git checkout – 文件名

用Git仓库中保存的文件,覆盖工作区中指定的文件

git reset HEAD 文件名

从暂存区移除对应的文件

git reset HEAD .

全部移除

移除文件
git rm -f 文件名

从git仓库和工作区中同时移除文件

git rm –cached 文件名

从git仓库中移除文件

忽略文件

创建一个名为.gitignore的配置文件,列出要忽略的文件的匹配模式

格式规范
  • 以#开头的是注释
  • 以/结尾的是目录
  • 以/开头防止递归(只关注于当前目录)
  • 以!开头表示取反
  • 可以使用glob模式进行文件和文件夹的匹配(glob->简化了的正则表达式)
glob模式
  • *表示匹配零个或多个任意字符
  • [abc]表示匹配任何一个列在方括号中的字符
  • ?只匹配一个任意字符
  • [a-z]表示匹配a到z范围内的字符
  • 两个星号表示匹配任意中间目录(比如a/**/z可以匹配a/z,a/b/z,a/b/c/z)

查看提交历史
git log

按时间先后顺序列出所有提交历史,最近的提交排在最上面

git lod -2

只展示最新的两条提交历史,数字可以按需填写

git log -2 –pretty=oneline

在一行上展示最近两条提交历史的信息

git log -2 –pretty=former:”%h | %an | %ar | %s”

在一行上展示最近两条提交历史的信息,并自定义输出的格式

回退到指定的版本
git log –pretty=oneline

在一行上展示所有的提交历史

git reset –hard

根据指定的提交ID回退到指定版本

git reflog –pretty=oneline

查看命令操作的历史

git reset –hard

再次根据最新的提交ID,跳转到最新版本

SSH

需要进行额外的配置,但配置成功后每次访问仓库不需要重复输入Github账号密码

SSH key

作用:实现本地仓库和Github之间免登录的加密数据传输

组成

  • id_rsa(私钥文件,存放于客户端电脑)
  • id_rsa.pub(公钥文件,需要配置到Github中)
生成SSH key

  • 打开Git Bash

  • 输入 ssh-keygen -t rsa -b 4096 -C “你的绑定邮箱”

  • 连续敲击三次回车,即可在 C:\Users\用户名文件夹.ssh文件夹中找到文件

配置SSH key
  • 打开id_rsa.pub文件,复制内容
  • 登录Github, 点击头像 -> Settings -> SSH and GPG Keys -> New SSH key
  • 将 id_rsa.pub 文件中的内容, 粘贴到 Key 对应的文本框中
  • 在 Title 文本框中任意填写一个名称, 用于标识Key的由来

Git分支

master(main)主分支

在初始化本地Git仓库时,Git默认帮我们创建了一个名字叫做master(main)的分支,通常称之为主分支

一般用于保存和记录整个项目已完成的功能代码

不允许直接在主分支上修改代码,因为这一行为容易导致整个项目崩溃

功能分支

指专门用来开发新功能的分支,是临时从主分支分叉出来的,当新功能开发且测试完毕后最终需要合并到主分支上
